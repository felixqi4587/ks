## 1. 目标与定位

为 1406 王国的一次真实 KvK「王城战」做的移动端协同网页。核心目标：**普通玩家只填一项（自己到王城的行军时间），打开主页就能通过一个 10× 动画 + 个性化"补兵时机尺"，直观知道整场仗怎么打、自己该几点做什么。** 其余全部由管理员配置，发布后即成为所有玩家打开网页看到的默认内容（无需额外链接）。

线上：https://felixqi4587.github.io/1406rocks/ ｜ 后端：Cloudflare Worker + KV（已存在，需小改）。

### 与 v5 的关系
v6 是对现有 v5（四标签：同步集结/炮塔分配/守城补兵/打法）的重构：**两个视图**取代四标签——玩家主页（唯一面向玩家）+ 管理员配置页（合并原三页，仅管理可见）。

## 2. 战斗模型（产品的事实基础，已经多源核验）

一个完整循环：

1. **我方错峰开集结**：少数大鲸鱼（攻城车）按错峰开集结，战力玩家加入。
2. **同步命中·拿下王城**：所有攻城车在同一秒（1–5 秒内）命中王城 → 假定必拿下（若同步攻城都拿不下，这场 KvK 已无解，工具无意义）。
3. **敌方来袭**：敌方鲸鱼反扑，通常也想同步，但没有协同工具 → 落地之间有 1–5 秒间隙。
4. **卡间隙 refill 守城**：我方在每次敌方命中**之后的间隙**把兵补满 → 满兵抵御。连环集结时：敌方第一辆车落地后立刻连续补兵、逐车补满（这正是连环功能存在的意义）。
5. 守住 → 回 3（敌人再来、再 refill）；失守 → 回 1（重组攻城）。

玩家"唯一在乎"的两个输出：**(A) 我方各攻城车的开集结顺序**（保证同秒命中）、**(B) 个人最佳补兵发兵时刻**（卡间隙）。

### 关键机制约束
- **集结准备时间统一 5:00**（双方；全王国约定都用 5:00，确保错峰同步成立）。PvP 建筑集结**到点才自动发车、不能手动提前**，所以同步靠"错峰开集结"。
- **集结（攻城车）行军不可加速**；**refill（守城补兵）行军可加速**。
- **refill 宁可晚一秒、不可早一秒**：路上的补兵不参与当前这次防守战斗（只有命中那刻城里的兵参战）；若补兵在敌方命中前、城内仍满员时到达 → 没有空位可填、被退回白跑（垃圾 refill）。最佳是**敌方每辆车落地后极短时间内到达**（落进 1–5 秒间隙），正好补上刚被打出的伤亡，满兵接下一车。
- 一人同一时间只能占一个结构（攻城/驻守/炮塔三选一），所以要分工。

### 角色与战时编制
- 战时全王国最强联盟合并成临时联盟；通常：**王城车一个联盟、炮塔车们一个联盟**。
- **大鲸**（少数，管理设）：开王城攻城车。
- **TG5 高兵营玩家**：加入王城车 + 负责守城 refill。
- **低兵营玩家**：加入炮塔车。
- **4 个炮塔集结者**：管理分配；**加入炮塔车是自由的**，玩家自己加。

## 3. 两个视图

### 3.1 玩家主页（唯一面向普通玩家）

唯一输入：**我到王城的行军时间（mm:ss，通常 1 分多，少超 2 分）**。

内容（从上到下）：

1. **唯一输入框**：我到王城行军。
2. **⚔️ 实时战场（10× 动态动画，循环 + 可拖拽时间条）** —— 主角。
   - 元素颜色：🟡大黄点 = 我方鲸鱼；🔴大红点 = 敌方鲸鱼；🟢小绿点（多个）= 我方加入者 + 守城补兵。
   - 方位由管理"我方相对王城方位"决定：我方在选定一侧，**敌方在对面一侧**（红点只从对面出现）。
   - 鲸鱼起始**距离 ∝ 行军时间**：越远 = 行军越久；**远的先开集结、行军更久，最后同时命中**。
   - 循环演：①错峰开集结 + 绿点加入 → ②同步命中·拿下（金色冲击波、城变金、绿色兵力条满）→ ③敌方来袭（红点带"落地 X.Xs"倒计时，绿条掉）→ ④卡间隙 refill（绿点群涌入、绿条补满、金光）→ ⑤守住，循环。
   - **可拖拽时间条**（⏸/▶ + 滑块）：玩家可手动慢放、定格看 refill 那一刻。
3. **🛡 补兵时机尺（答案面板，必须跟着玩家行军变）** —— 核心。
   - 每个敌方鲸鱼一行；玩家改自己的行军，每行的大字实时变。
   - 每行给一个醒目大字：例如「敌A → 他**集结剩 0:08** 时发兵」；敌人比你慢/快自动切换说法（看集结剩余 or 看来袭落地倒计时）。
   - 玩家挑当场正在来袭的那条敌鲸，照大字发兵。
4. **你该做什么（进攻指引）**：TG5 步兵营 → 加入【王城车·鲸X】（显示发车顺序）；否则 → 加入【炮塔车】（自由加入）。
5. **战斗指南（替代"复制战报"）**：内置清晰的分轮行动指南——
   - **进攻轮**：加入分配给你的车 → 看动画/发车顺序 → 同步命中；攻击阵容默认 **步50/骑20/弓30**（敌步兵墙多则切 30/20/50）。
   - **防守轮**：拿下后**立刻换防御英雄 + 改比例 步60/骑40**；按"补兵时机尺"在敌每车落地瞬间补满；最强号常驻当驻防队长。
   - **通用提醒**：开打前后多刷新本网站看时间；改了兵种/英雄要回来重看自己的行军/发兵时刻。
6. **可选战术提示**：若管理填了"战术安排&加入要求"，主页在一处显示对应提示/入口；**没填则不显示**。
7. **折叠收起**：炮塔分配（4 个集结者是谁）、战术安排全文。

### 3.2 管理员配置页（合并·仅管理可见）

进入：点主页 🔒 → 输共用密码 + 自己的名字（见 §6，浏览器记住、可退出）→ 解锁。原"同步集结/炮塔分配/守城补兵"三页合并为这一页，6 块：

1. **我方相对王城方位**：左侧/右侧（左显示在左、右显示在右）。同时决定动画里我方/敌方在哪边。
2. **战术安排 & 加入要求（可选，合并一个文本框）**：填了才在主页显示提示。
3. **王城攻城车（大鲸）**：可加/删多条，每条 = 名字 + 到王城行军（mm:ss）+ 加入者。加入者填写方式 = **输名字→＋加入→生成可×删除的名字标签**，或在"或写要求"框直接写要求（如"TG5 步兵营"）。**每条鲸鱼的加入者区措辞/布局必须完全一致。** 下方"发车顺序预览"显示自动算出的开集结顺序（给鲸鱼照着开）。
4. **炮塔分配（4 个集结者）**：4 行，各填谁开该炮塔车。无第 5 个王城。
5. **敌方鲸鱼（守城补兵基准）**：可加/删多条，每条 = 名字 + 到王城行军（mm:ss）。
6. **发布到全王国**：POST 到后端（含乐观锁版本校验）；显示"当前版本：<名字> 于 <时间> 发布"。

每块配小字 = **给管理的填写说明**（说明这是什么、怎么填），不暴露内部算法措辞。

## 4. 计算公式

记号：行军/时间单位为秒。集结准备 `GATHER = 300`（5:00，双方固定常量）。补兵落地后安全余量 `DELTA ≈ 2`（秒，落进间隙的小冗余，代码常量）。

### 4.1 攻城错峰（A：开集结顺序）
设各攻城车到王城行军为 `march_i`：
```
base   = max(march_i)                 // 最远车
delay_i = base - march_i              // 该车相对"最远车开集结"的延迟；最远车 delay=0 最先开
// 全部命中时刻（相对最远车开集结）= GATHER + base，对所有车相同
```
主页/配置页显示：最远车先开 → 其余按 `delay_i` 依次开 → 同时命中。建筑集结到点自动发、不可提前。
**输入校验**：若有非最远车被算成最先开（说明某车行军填错），就近内联提示"行军可能填错了"，防止整条同步被带偏。

### 4.2 守城 refill（B：个人发兵时刻，逐个敌鲸）
设某敌鲸到王城行军 `Em`，我方补兵行军 `R`（可加速 → R 为玩家实际/加速后行军）：
```
enemy_landing  = enemy_open + GATHER + Em          // 敌方命中时刻
target_arrive  = enemy_landing + DELTA             // 目标：落地后极短时间到（落进间隙）
send           = target_arrive - R                 // 我方发兵时刻
// 下限保护：补兵到达必须 ≥ enemy_landing（不可早于落地，否则满城被退回）

// 以可观测锚点表达（防守方两种都看得到）：
gather_remaining_at_send = R - Em - DELTA
if gather_remaining_at_send > 0:
    提示「当敌X集结剩 (R-Em-DELTA) 时发兵」          // 你比该敌鲸慢：他攻城队还没出发你就得发
else:
    提示「敌X发车后、当来袭落地倒计时 = (R-DELTA) 时发兵」  // 你比该敌鲸快：等他发车后看来袭倒计时
```
补兵答案面板对每个敌鲸独立计算，随玩家 `R` 实时更新。

## 5. 数据模型（存于 Cloudflare KV，单键 `plan`）

```json
{
  "v": 6,
  "side": "right",                       // "left" | "right"，也驱动动画方位
  "tactics": "",                          // 可选合并文本；"" 表示不显示
  "castleWhales": [
    { "name": "鲸a", "mm": 1, "ss": 30, "joiners": ["张三","李四"], "requirement": "TG5 步兵营" }
  ],
  "turretLeaders": ["孙七","周八","吴九","郑十"],   // 固定 4 个，可为空字符串
  "enemyWhales": [
    { "name": "敌A", "mm": 1, "ss": 10 }
  ],
  "updatedAt": "2026-06-17T14:28:00.000Z",  // 服务端写
  "updatedBy": "chris"                        // 服务端写（管理登录时填的名字，随 POST 提交）
}
```
`GATHER`、`DELTA` 为前端常量，不入库。`joiners` 与 `requirement` 同时存在，管理用哪个填哪个；玩家端两者都显示（有则显示）。

## 6. 鉴权：共用密码 + 单独名字（浏览器记住 + 可退出）

- **方案（已选）**：一个**全咸鱼小队共用的搞笑短密码**（服务端常量）+ 管理在登录时**单独填一次自己的名字**（仅作署名）。不再用"密码+名字合一"。
- **共用密码**：默认 `666`（先沿用；服务端 `ADMIN_PASS` 一处改即可，将来想换搞笑密码随时改）。
- **流程**：点 🔒 → 输共用密码 + 名字 → 校验通过后**存浏览器 localStorage**；此后本浏览器进管理**不再要求输入**。提供**退出**按钮清空（用于换人/换密码）。
- **后端校验**：`POST` body `{ password, name, ... }`；`password === ADMIN_PASS` 才接受；`updatedBy = name`。
- 说明：这不是真正的安全（共用密码、前端行为可见），只为防误改并记录"谁发布的"。

## 7. 并发：乐观锁 + 版本校验

- 每次 `GET /plan` 返回 `updatedAt`。管理进入编辑时记住该 `baseUpdatedAt`。
- `POST /plan` 带 `baseUpdatedAt`：
  - 若库中已有计划且其 `updatedAt !== baseUpdatedAt` → 返回 **409 冲突**，附当前最新计划。
  - 否则写入，更新 `updatedAt`（服务器时间）、`updatedBy`，返回 ok。
- 前端遇 409：提示「<updatedBy> 于 <updatedAt> 刚发布了新版，请**刷新后再改**；或**仍要覆盖**」。选覆盖则重新 GET 取最新 `updatedAt` 作为 `baseUpdatedAt` 再 POST。

## 8. 后端改动（Cloudflare Worker，现有 1406rocks-plan）

现状：`GET /plan` 公开读；`POST /plan` body `{password,plan}`，校验 `password === "666"`，存 KV。需改：
1. 密码校验改为 `password === ADMIN_PASS`（env，默认 `666`）；`updatedBy = body.name`（登录时填的名字，随 POST 提交）。
2. `POST` 接收 `baseUpdatedAt`，做乐观锁：与库中 `updatedAt` 不一致则 409（附 `current`）。
3. 存储记录加 `updatedBy`；`GET` 一并返回 `updatedAt`、`updatedBy`。
4. KV、CORS、workers.dev 子域沿用现有（`https://1406rocks-plan.kingshot1406.workers.dev`）。

## 9. 动画规格（实现要点）

- SVG + requestAnimationFrame；颜色用十六进制（不在 SVG 里用 CSS var）。
- 主时钟 `t` 循环；`render(t)` 为 t 的纯函数（便于拖拽 scrubber 任意定位）。
- 拖拽时间条：`<input type=range>` 绑定 t；拖动暂停自动播放；⏸/▶ 切换。
- 方位由 `side` 决定我方一侧、敌方对面。
- 鲸鱼起始距离 ∝ 行军；远的先开集结环（gather ring 填充）、行军更久；同步命中冲击波。
- 兵力条（绿）：命中掉、refill 补满；refill 为多个小绿点从我方兵营涌入。
- 补兵时机尺为独立面板（HTML），按 §4.2 随玩家行军实时渲染。

## 10. 非目标 / 范围外

- **不做任何"实时监控"类功能**（宏观占领仪表盘、炮塔重置实时计时器等）：这些要战时动态数据、没人会在网页上实时更新。本工具定位 = **赛前战略部署 + 个人时机计算 + 行动指南**，实时数据玩家在游戏里看。
- 不做"喊话员直播触发"：玩家盯游戏内真实的敌方集结/来袭倒计时，照个人数字执行，无需任何人点按钮。
- 不做绝对 UTC 排程（除现有时钟/阶段提示）。
- 炮塔加成数值、占领计分、胜负条件仅作参考说明（折叠区/打法），不参与计算。
- 不做玩家账号体系；管理身份仅靠共用密码 + 登录时填的名字。
- 不嵌入任何游戏美术/截图；保持非官方、与 Century Games 无关。

## 11. 假设与待确认

- `DELTA`（落地后余量）取 ~2 秒；如实战发现间隙更大/更小可调常量。
- 补兵答案面板默认列出所有敌鲸；玩家自行挑当场来袭那条（不需要选择器）。
- 攻城车的"加入者名单/要求"仅展示给玩家参考，不做准入校验。
- 现有前端 v5 的 Ursa 暖炭+琥珀视觉与 sigil 沿用。

## 12. 战略审计结论（26 agent 已核验，作为定稿依据）

- **攻城错峰公式 = 行业标准**（kingshot.net / kingshotlab 等同款），无需改。
- **守城补兵计算器是独一份**（所有现有工具只做进攻）——本工具最大差异化、核心护城河。
- **胜负 = 3h 连占 / 6h 累计、按联盟**——官方 KvK（Kingdom of Power）已确认（"2.5h/5h"是另一普通服模式，不用）。
- **守城"落地后瞬间补满"模型成立**：路上补兵不参战、满员到达=无空位被退回，正确（用户原判断无误，保留不改）。
- **明确不做**：硬性三车计时阶梯（来源不足、会重复计算 march）、"打炮塔才能开消耗车"门槛（机制不成立）、实时监控仪表盘/炮塔实时计时器（需战时动态数据、不切实际）。
- 兵种/英雄默认值（写入分轮指南，仅建议非强制）：攻击 步50/骑20/弓30（敌步墙→30/20/50，固定不变）；守城 步60/骑40 + 防御英雄；最强号常驻驻防队长。
- 玩家主页**不显示个人开车时刻**（普通玩家只加入、不开车）；进攻信息 = 发车顺序 + 每车加入者名单/要求；玩家行军只驱动守城补兵时机尺。
